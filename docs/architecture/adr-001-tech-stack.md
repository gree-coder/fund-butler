# ADR-001: 技术栈选型决策

## 状态
已接受 (Accepted)

## 背景
基金管家项目需要选择一套稳定、高效、易于维护的技术栈，支持基金数据查询、持仓管理、收益分析等核心功能。

## 决策

### 后端技术栈
| 组件 | 选型 | 版本 | 理由 |
|------|------|------|------|
| 框架 | Spring Boot | 3.4.3 | 生态成熟，企业级支持 |
| 语言 | Java | 17 | LTS 版本，性能优化 |
| ORM | MyBatis-Plus | 3.5.9 | 简化 CRUD，代码生成 |
| 数据库 | MySQL | 8.0 | 关系型数据，事务支持 |
| 缓存 | Caffeine | - | 本地缓存，MVP 阶段足够 |
| HTTP 客户端 | OkHttp | 4.12.0 | 外部 API 调用 |

### 前端技术栈
| 组件 | 选型 | 版本 | 理由 |
|------|------|------|------|
| 框架 | React | 18 | 组件化，生态丰富 |
| 语言 | TypeScript | - | 类型安全，IDE 友好 |
| 构建工具 | Vite | - | 快速启动，HMR |
| UI 组件库 | Ant Design | 5 | 企业级，金融数据友好 |
| 图表库 | ECharts | 5 | 金融图表支持完善 |
| 状态管理 | Zustand | - | 轻量级，API 简洁 |

## 后果

### 正面
- 技术栈成熟稳定，社区支持好
- 前后端分离，团队可并行开发
- TypeScript 提供类型安全，减少运行时错误

### 负面
- 需要维护两套技术栈的知识
- 前端构建工具链较复杂

## 替代方案考虑

| 方案 | 放弃原因 |
|------|---------|
| Vue 3 + Vite | React 生态更丰富，金融组件库更多 |
| PostgreSQL | 团队更熟悉 MySQL，运维成本低 |
| Redis 缓存 | MVP 阶段数据量小，Caffeine 足够 |
